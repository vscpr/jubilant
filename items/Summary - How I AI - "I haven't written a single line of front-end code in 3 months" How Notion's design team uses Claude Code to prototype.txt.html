Designs Should Encounter Reality as Early as Possible

Brian Lovin was designing conversations in Figma. As a designer on Notion AI, he would mock up golden paths — the user says this, the AI responds with that, a page gets created perfectly — and hand them to engineers. Then the engineers would build it and, inevitably, it wouldn't work that way. The AI would get stuck, ask follow-up questions, do the wrong thing entirely. The chat experience he'd designed in static mockups bore almost no resemblance to what it felt like to actually use the product.

That gap between design artifact and lived reality is what drove Lovin to build something called Prototype Playground — a shared Next.js repository where Notion's design team can vibe-code working prototypes using Claude Code, connected to real AI models, in a real browser environment. It's a deceptively simple idea that reveals a deeper shift in how design teams will need to work as more products incorporate AI.

The Architecture of Prototype Playground

Prototype Playground is, as Lovin repeatedly emphasizes, "nothing magical." It's a Next.js app deployed on Vercel. Inside the app directory, every designer (or PM, or engineer) gets their own namespace — a folder with their name. Every subfolder within that is a standalone prototype. There's no global layout, no rigid structure to adhere to. Each prototype is its own self-contained page.

The front end renders a simple index: a chronological list of prototypes, organized by who's been working on what recently. The power comes from two dimensions. First, visibility — everyone can see what everyone else is building, which sparks ideas and cross-pollination. Second, portability — because the code lives in one repo, anyone can "yoink" techniques or components from a teammate's prototype and incorporate them into their own, usually by just telling Claude to do it.

Before Prototype Playground existed, several Notion designers already prototyped in code. The problem was fragmentation. Everyone maintained their own repository, their own Next.js instance, rebuilding shared patterns from scratch. If someone wanted to make something look "Notion-y," they were reinventing the wheel each time. The playground consolidates all of that with shared components, shared styles, a library of Notion's colors, typography, and icons, plus templates for common starting points like a standard Notion sidebar layout.

There's no database backing any of this. It's all files on disk — metadata JSON files that get collated to render the homepage list, prototype files with actual code, and that's it. Push to GitHub, deploy to Vercel. The simplicity is the point.

The Workflow: Plan, Talk, Verify

Lovin's actual prototyping workflow reveals a set of principles that any practitioner can learn from. He works in a three-pane setup: Claude Code in the terminal on one side, Cursor showing the code changes in the middle, and the browser output on the right. He uses a keyboard shortcut (Caps Lock + G) to snap these into position instantly.

His first principle: always plan before building. He switches Claude Code into plan mode before writing any code. "For whatever reason, you just get better outcomes," he says. The key is actually reading the plan — this is where having a development background gives you an edge, because you can spot when something in the plan doesn't look right. But even without that background, the structure of planning first produces better results.

His second principle: talk instead of type. He uses a tool called Monologue that converts speech to text, letting him dictate prompts conversationally rather than typing them out. It's faster and, importantly, tends to produce more natural, detailed descriptions of what he wants.

His third and most important principle is what might be called the automation imperative: anytime the AI asks you to do something, try to teach it to do that thing for itself. This is the driving philosophy behind every optimization in his setup. Claude used to say "I've implemented your feature, go take a look and let me know what you think." That meant Lovin had to switch to the browser, visually inspect everything, and report back. Instead, he taught Claude to launch Chrome via MCP, navigate to the prototype, click buttons, check for visual correctness, and verify functionality — all autonomously.

The same principle applies to linting. Instead of letting Claude produce code that might have errors you only discover in the browser, he configured it to run ESLint and check for TypeScript errors automatically after every change. The result: longer autonomous runs with fewer interruptions and better outcomes.

Figma to Code in 27 Seconds

The most visually impressive demonstration is the /figma slash command. You paste a link to a Figma frame, and Claude extracts the design, implements it in code, then enters a verification loop — opening the browser, comparing the implementation to the original Figma file, fixing discrepancies, and repeating until it's either matched or completed three iterations with no further changes.

Lovin estimates the raw AI gets designs about 60% correct. But the slash command — with its structured phases, forced MCP invocations, and automated verification loop — pushes that to roughly 80%. The remaining 20% involves things like missing hover states or broken images, which are easy follow-up prompts.

The broader implication, as host Claire Vo observes, is staggering for the industry: "Think about how many engineers sit there and pixel-pull over Figma prototypes into the front end. For my entire career, 20-plus years in tech, these asset-to-asset handoffs have been the most expensive part of implementing something." What used to be the costliest friction point in product development now takes seconds to scaffold, freeing time for the work that actually matters — optimization, performance, how the product feels.

Teaching Claude to Fish: Skills and Slash Commands

Lovin has built a small ecosystem of Claude Code customizations that make Prototype Playground usable by the broader team. These break down into several categories.

Claude.md and Claude.local.md files provide layered context. The global Claude.md file, committed to the repo, contains instructions about tooling (Bun, Tailwind), project structure, and shared conventions. The local file, gitignored and personal to each machine, adds context like the user's Prototype Playground username and their working directory, plus guardrails like "don't mess with other people's files."

Slash commands are glorified prompts that can also run scripts. The /create-prototype command scaffolds a new prototype with the correct file structure, metadata, and boilerplate — no manual file creation needed. Critically, the command includes example code snippets that show Claude what success looks like. "AI is better with good context, but it's also really, really good if you just provide examples of how to do things," Lovin explains.

The /deploy command is Lovin's most ambitious attempt at lowering the barrier for non-technical designers. It checks prerequisites (GitHub CLI installed, authenticated), creates a Git branch, stages changes, commits, pushes, opens the PR in the user's browser, then monitors CI checks every 30–60 seconds — automatically fixing any failures and re-pushing until all checks pass. The entire Git workflow, from branch creation to green CI, happens without the user needing to understand any of it. Yet everything is communicated in plain English so curious designers can learn by watching.

Skills are capabilities Claude detects and uses automatically. The find-icon skill solves a pervasive problem: AI models hallucinate icon names. They'll assume a search icon is called "search" when the codebase calls it "magnifying glass." The skill includes a TypeScript script that searches through Notion's 5,000+ icons using synonyms and related terms. Claude invokes it automatically whenever it needs to implement an icon, dramatically reducing import errors on the first pass.

Why Designers Need to Understand Models

Lovin still spends 60–70% of his time in Figma. For many design tasks, static mockups are perfectly adequate — you can look at them and know they're roughly right. But for anything involving AI, that breaks down completely.

You can design what a chat input looks like in Figma. You can draw a chat bubble, a send button, a model picker dropdown. What you cannot design in Figma is what it actually feels like to use that thing. What happens when the model thinks for two minutes and the user stares at an empty screen? What happens when it hits an error? What should incremental progress look like? These questions can only be answered by building with real models in real code.

This creates an uncomfortable reality for the design profession. As Lovin puts it, "every month, the whole industry has to learn what the new capabilities of this model are. And then a month later, it's all irrelevant because the new thing has come out." It feels wasteful, but it's necessary. Designers building AI products must understand what models can and cannot do in order to design experiences that live "right at the edge of what the model is going to be able to do well." Designing beyond that edge produces products that break in production. Designing too far inside it leaves capability on the table.

The Quality of Your Inputs

When asked about his prompting technique for when AI isn't cooperating, Lovin's answer is disarmingly human: go to bed. "There's a direct correlation with how good of things I can make and how tired I am," he says. When Claude seems to be doing terrible work, he goes back and rereads what he actually wrote and realizes he made no sense. The quality of the output is directly correlated with the quality of the context you give it. Sleepy, tired, lazy "please fix this" commands produce bad work.

It's a reminder that in the age of AI-assisted everything, the bottleneck isn't the model. It's the clarity of thought you bring to the conversation.

What To Do With This

  • Set up a shared prototyping repository for your design team using a simple Next.js app with namespaced directories per designer. Consolidating prototypes in one repo creates visibility into what teammates are building and lets everyone reuse components, styles, and clever techniques — eliminating the fragmentation of everyone maintaining separate repos.
  • Create a Claude.md file at your project root with tooling preferences, project structure, and conventions, plus a gitignored Claude.local.md for per-user context. This layered approach gives AI assistants consistent shared context while allowing individual customization like usernames and working directories.
  • Switch Claude Code to plan mode before starting any prototype and actually read the plan before proceeding. Planning first produces measurably better outcomes, and reviewing the plan catches structural errors before any code is written.
  • Apply the automation imperative: every time Claude asks you to do something, ask whether you can teach it to do that thing itself. This compounds over time — each automation means longer autonomous runs, fewer interruptions, and better end results. Start with having Claude lint its own output and check its work in a browser via MCP.
  • Build slash commands that include example code snippets showing what success looks like, not just instructions. AI performs significantly better with concrete examples than with abstract descriptions. Provide sample output files, boilerplate, and templates directly in the command definition.
  • Create a /deploy or similar end-to-end command that handles the full Git workflow — branching, committing, pushing, PR creation, and CI monitoring. This dramatically lowers the barrier for non-technical team members while still exposing them to Git concepts through plain-English narration of each step.
  • Use Claude Skills paired with TypeScript scripts for recurring problems like icon name hallucination. The combination of natural language prompting in the skill's markdown file with programmatic tools in bundled scripts is far more effective than either approach alone.
  • Connect the Figma MCP and build a structured command that extracts designs, implements them, then enters a browser-based verification loop comparing implementation to the original. The automated comparison loop can push accuracy from roughly 60% to 80% on first pass, with remaining issues easily addressed in follow-up prompts.
  • For AI product design specifically, prototype with real models in real code rather than mocking golden paths in Figma. Static mockups cannot capture model latency, error states, follow-up questions, or the fundamental unpredictability of AI responses — all of which critically shape the user experience.
  • When your AI outputs feel low quality, check your inputs before blaming the model — and consider whether you're too fatigued to prompt well. The correlation between input clarity and output quality is direct and unforgiving. A fresh mind writing a detailed, thoughtful prompt will outperform a tired mind using the most powerful model.
