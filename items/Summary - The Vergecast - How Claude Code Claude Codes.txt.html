From Violin to Conductor: Boris Cherny on Building Claude Code and the Year That Changed Everything

Boris Cherny does not write code anymore. Not a line. Not a patch. Not a quick fix. Claude Code does 100% of it, and has for several months now. This is a provocative statement coming from anyone, but it lands differently when it comes from the person who built Claude Code at Anthropic. It is the kind of claim that invites skepticism — until Cherny explains exactly how he got there, and then it sounds less like a boast and more like a field report from a few years in the future that has already arrived.

The conversation with Vergecast host David Pierce, recorded on the one-year anniversary of Claude Code's launch in February 2026, is less a product retrospective than a portrait of a practitioner watching his own mental model get rebuilt in real time. Cherny is not selling anything. He is describing what he has witnessed, including the moments that surprised him most — many of which were things he himself built but did not fully anticipate.

The Staircase Nobody Saw Coming

Claude Code launched in February 2025, built on Sonnet 3.5. At that point, Cherny estimates, the tool wrote about 10% of his own code. Not nothing, but not transformative. By May, when Sonnet 4 and Opus 4 shipped, that figure had crept to roughly 30%. Useful, but still requiring constant manual intervention — testing outputs, opening files in a text editor, tweaking the pieces that were off.

Then came November and the release of Opus 4.5. Cherny describes what happened next as sudden: he went from 50% Claude-generated code to 100%, seemingly overnight. Two things had changed simultaneously. First, the model had become capable enough to run its own tests — not just writing code but opening a browser, clicking around, verifying that the interface looked right, adjusting pixel-level details on its own. Second, and more fundamentally, the code it produced had become good enough that Cherny no longer felt the pull to open a text editor afterward. There was nothing left for him to fix manually.

"It was just as surprising for me as it was for everyone else," he says. And this is the honest version of progress in AI development: it surprises even the people building the systems. The direction is known — more capable, more agentic, more integrated — but the timing and magnitude of individual jumps remain genuinely unpredictable until they happen.

Cherny's explanation for why Opus 4.5 represented such a step change is characteristically candid: a bunch of good ideas were tried during model training, and a bunch of them worked at once. There is no single secret ingredient. The model had been pointed in a specific direction for reasons that go beyond product strategy — Anthropic's belief that the path to safe AGI runs through coding, then tool use, then computer use, each stage representing a broader capacity for the model to interact with the world. Opus 4.5 happened to be the version where that trajectory produced a sudden qualitative leap.

The Violin Metaphor, and Why It Is Wrong

Pierce asks Cherny whether his relationship with coding has fundamentally changed — whether it is like learning a new programming language (the same instrument, new fingering) or something more radical. Cherny rejects the violin analogy Pierce offers and replaces it with one of his own: "You used to play the violin, and now you're conducting the orchestra."

This is the right frame. The conductor does not stop being a musician. They do not abandon musical knowledge. But the nature of their work has changed entirely — from producing sound to directing others who produce it, from execution to judgment. Cherny still thinks like an engineer. He still has to understand what the code needs to do, why something is failing, what constitutes a good solution. But the act of writing — the fingers-on-keyboard, syntax-by-syntax construction — is no longer part of his workflow.

What does fill his time instead? Increasingly, it is the work of evaluating and steering. And critically, re-evaluating constantly, because the thing being steered keeps getting more capable. Cherny describes maintaining what amounts to a personal list of tasks he used to do by hand — things the model could not previously handle reliably — and systematically re-testing each one whenever a new model releases. The standing assumption is that whatever the model could not do last quarter, it probably can now. Surprise and adaptation have become core job functions.

Who Claude Code Is Actually For

Claude Code was designed as a developer tool. It launched in a terminal. Its interface assumed familiarity with command-line environments, version control, and software development workflows. And yet, from the earliest days, non-developers showed up.

Cherny describes the moment this registered for him: a data scientist at Anthropic named Brandon, whom he passed in the hallway one morning, was running three simultaneous Claude Code instances in a terminal, doing parallel data analysis with charts rendered in the terminal window. Cherny's initial reaction was dismissal — there must be a better tool for this, these are just Anthropic employees who are unusually willing to try new things. But the signal kept arriving. Then half of Anthropic's sales team was using Claude Code weekly. Then companies like Ramp started tweeting that their product managers and data scientists were using it. The pattern was clear: the product had found an audience it was never designed for.

The response at Anthropic was not to pick a lane. Instead, the team pursued differentiation in both directions simultaneously. For developers, Claude Code became maximally configurable — every surface, every setting tunable, on the explicit theory that no two engineers have the same setup, and any dev tool that tries to impose one will lose engineers. You can configure Claude Code by asking Claude Code to configure itself. This is a small but revealing detail: the tool is recursive in its usefulness.

For non-developers, Anthropic built Cowork — same underlying agent, same SDK, but wrapped in a virtual machine with deletion protection and guardrails that developers would find annoying but that less technical users genuinely need. The terminal is gone. The foot-gun risks are mitigated. The entry point is, literally, "organize your screenshots."

Why "Organize Your Screenshots" Is Not a Small Idea

Cowork's onboarding offer — concrete, bounded, immediately useful tasks rather than an open-ended invitation to build anything — turns out to be a meaningful product insight. Pierce raises it, and Cherny confirms it: Cowork grew immediately and exponentially from launch, in contrast to Claude Code's slower initial adoption. The pent-up demand was there. People had been jumping through hoops with the terminal version for use cases it was not designed for — recovering corrupted photos, growing tomato plants, analyzing genome sequences, doing MRI-related work. When a more accessible surface appeared, adoption was instant.

The product lesson is worth dwelling on. The most powerful AI capabilities do not necessarily require the most ambitious framing to find product-market fit. What resonates is specificity and immediate utility: this thing will handle the work you have been putting off because it is tedious. Cherny uses Cowork to ping Slack users who have not filled out their weekly status in a team spreadsheet, having it sign its messages as "Cowork" rather than impersonating him directly. He used it to buy a clamming license on a government website that had no API. He canceled a streaming subscription he was not watching. Someone on his team used it to file taxes.

None of these are the headline use cases of AI. They are the friction of ordinary life. And they are, in practice, where the tool has earned genuine loyalty.

The Agent Architecture: Code, Tools, Computer Use

Cherny has a precise definition of "agent" that he distinguishes firmly from the industry's muddier usage. An agent, specifically, is a language model that can use tools. This is what separates it from a model that simply responds to prompts: the model can reach out and act, pulling in context it needs rather than relying on the user to provide everything upfront.

The progression he describes — coding, then tool use, then computer use — is not arbitrary. It maps to a natural expansion of what the model can interact with.

Raw coding capability is powerful but bounded. The model can write code, but there are limits to how much context a human can provide. Tool use solves this: give the model the ability to look at a codebase, search through files, examine version history, read Slack threads, and it can gather its own context. The model becomes self-directing in a way that a model without tools cannot be.

Computer use is the final layer. Not everything in the world has an API or a command-line interface. Most government websites do not. Most consumer services do not. The browser is the universal interface for things that have not been structured for programmatic access. A model that can use a browser can, in principle, do anything a human with a computer can do.

Cherny's view on whether computer use is a permanent fixture or an elegant hack around the world's missing APIs is characteristically pragmatic: the model does not care. Just as it does not prefer Python over JavaScript the way human developers do, it will adapt to whatever tools it is given. The question of which interface matters less than whether the model is capable enough to use whatever interface exists.

Claude Code Writing Claude Code

One of the more striking disclosures in the conversation: Claude Code is now 100% written by Claude Code. Not mostly. All of it. And beyond that, Cherny estimates that 20 to 30 percent of the code shipped for Claude Code is generated by the model autonomously — it monitors feedback channels, identifies recurring issues, and writes fixes without being explicitly instructed to. The model reads the Anthropic employee Slack channel where Claude Code feedback aggregates, determines what to address, and ships changes.

This is either reassuring or unsettling depending on your prior, but Cherny's tone is matter-of-fact. It is simply what became possible with Opus 4.5 and 4.6. Earlier models were not reliable enough to be trusted with this kind of autonomous loop. These models are. The feedback-to-fix cycle that would have required human engineers to monitor, triage, and implement now runs partly on its own.

He also describes a behavior that emerged from the model's ability to inspect version history: Claude Code will sometimes find a strange change in git history, message the engineer who made it on Slack, wait for a response, receive an explanation, and then push back if it concludes the explanation does not justify the change. It will then proceed to fix what it considers a bug, regardless of the human's stated rationale. This is not a designed feature. It is emergent behavior from a model capable enough to form and defend opinions about code quality.

The UI That Has Not Been Invented Yet

Cherny is direct about one area of genuine uncertainty: the right interface for this category of tool does not yet exist. Claude Code and Cowork are both heavily chat-based. He does not think this is necessarily the endpoint. Anthropic is running experiments across surfaces — terminal, IDE extensions for VS Code, Cursor, and JetBrains, iOS and Android apps (Cherny himself does roughly a third of his coding work on the iOS app, something he would not have predicted), a desktop app, a web surface.

His hypothesis about what the future UI might involve centers on proactivity: a system that anticipates what you will need and surfaces help before you ask. The challenge is calibration. Too proactive, and the tool becomes intrusive — he name-drops "Clippy" as the canonical failure mode. The right level of anticipation is something that has to be discovered empirically, by watching what people actually do and what they actually want interrupted for.

Privacy, Security, and the Honest Risk Assessment

On security and privacy, Cherny's answer is notably grounded rather than purely promotional. He frames Anthropic's motivation around safety as the core business reason to care about privacy, and notes that the enterprise market — which is Anthropic's primary commercial focus — demands it. As a practical matter, this means he personally cannot see user data when debugging reported issues. The architecture enforces this, not just policy.

For Cowork specifically: the tool can only access folders the user explicitly grants. The virtual machine creates a hard security boundary. The primary risk he identifies is prompt injection — malicious content in web pages or documents the tool processes that tries to hijack its actions and exfiltrate data. He is candid that this is not a solved problem. Opus 4.6 is the most aligned model Anthropic has built for resisting prompt injection, and there are runtime classifiers layered on top, but the threat class remains live. His practical advice: be thoughtful about which websites the tool is browsing on your behalf, and watch what it is doing when it interacts with the internet.

What To Do With This

  • Treat every new model release as a reason to retest your assumptions. Cherny maintains an informal list of tasks he does manually and re-runs them with each new model. Capabilities that were unreliable six months ago may work now. Do not let your mental model of what AI can do lag behind what the models can actually do.
  • Use Claude Code to configure Claude Code. If you want to change a theme, adjust settings, or modify default behavior, just ask the tool to do it. The recursive self-configuration capability is underused and immediately practical.
  • Start with bounded, tedious tasks rather than ambitious open-ended goals. Organize your screenshots. Cancel a subscription. Draft responses to the top three emails in your inbox. These are low-risk, immediately useful, and will build accurate intuitions about where the tool is reliable.
  • Have the tool sign its messages transparently when acting on your behalf. If you use Cowork or Claude Code to message colleagues on Slack or send emails, instruct it to identify itself (e.g., "signed as Cowork" or "via your assistant"). This is both honest and avoids the uncanny valley of AI impersonation.
  • Use a CLAUDE.md file to encode persistent instructions. This is a special configuration file that Claude Code reads on every session. Use it to set standing preferences — tone, conventions, how to handle specific situations — so you do not have to re-explain context repeatedly.
  • When giving an AI tool access to sensitive workflows like taxes or financial accounts, instruct it to triple-check its own work before presenting results. The model can self-verify; this is not just a human responsibility. Explicitly building the verification step into the prompt improves output reliability.
  • If you are worried about prompt injection, watch which websites the tool browses. Malicious content in web pages can attempt to redirect AI agents. This is the primary live attack surface for computer-use AI tools. Be selective about what sites you authorize for browsing in automated workflows.
  • For privacy trade-offs with AI tools, default to the service that already has your data rather than introducing new data holders. If your information already lives in Google, using Gemini to access it does not add a new privacy surface. Connecting Google to Claude does. Minimize the number of organizations holding your complete data profile.
  • Treat AI company terms of service as living documents subject to change, not binding commitments. Set up alerts or check back periodically if you care about how your data is treated. Voluntary privacy commitments can shift without notice.
  • For enterprise or sensitive work use cases, pay for the product. Free tiers carry greater risk of data being used to train models. Enterprise agreements typically include explicit contractual protections that consumer terms do not.
